# Steven S1 安全边界与遗留风险 v1

- 日期：2026-07-17
- 适用范围：本机独立脱敏 Demo
- 当前结论：没有阻断 S1 本机 Demo 验收的问题；存在阻断真实 Key、D2、受控试运行和生产化的安全前置项。

## 1. 已保持的安全边界

- 浏览器只通过 `https://localhost:15443` 访问；Caddy、Next.js、FastAPI 只监听 loopback。
- Session Cookie 保持 `Secure`、`HttpOnly`、`SameSite=Lax`、`Path=/`。
- CSRF Cookie 保持 Secure，所有状态变更继续要求合法 CSRF Token 与 Origin/Referer。
- S1 沿用 permission-based RBAC，不信任前端提交的 actor、角色、审批状态、版本号、文件路径或审计字段。
- 四种 legacy 身份头继续由 FastAPI 与 Next.js proxy 拒绝。
- 提交人不得审批自己的事项。
- staging/production 的 session 模式不得退回 InMemory S1 Repository 或内存审计。
- 正式 Word 仅允许已批准事项生成；历史版本不得覆盖、删除或复用版本号。

## 2. Word、模板与附件安全

已落实：

- Demo 模板只含脱敏内容和 `.invalid` 联系方式。
- 模板变量使用明确的 `{{variable_name}}`。
- 未替换变量进入 `draft_error` 并阻断提交和正式导出。
- 文件名与 storage key 由服务端生成，客户端不能指定正式路径。
- Word 生成后重新打开校验并计算 hash/size。
- 本轮不支持任意模板上传、宏文档、外部链接抓取、OLE、PDF、邮件或打印。

后续若开放模板或附件上传，必须新增：

- `.docx` ZIP/XML 安全解析、压缩炸弹和大小限制。
- 宏、外部关系、OLE/嵌入对象、远程图片和路径穿越检测。
- 上传文件隔离、恶意文件扫描、MIME/扩展名双重校验。
- 模板启用审批和版本冻结。

## 3. OCR/AI 边界

- S1 OCR purpose 仅预留 `tender_source_extraction`。
- S1 AI purpose 仅预留 `clause_draft`。
- OCR/AI 输出只能进入 `review_candidates(needs_review)`。
- 人工确认前不得写 `steven_tender_*` 正式表。
- AI 不得决定模板启用、供应商、预算、审批或正式导出。
- 本轮没有调用 Azure、DeepSeek 或其他 Provider，没有读取或注入真实 Key。

## 4. 风险分级

### P0：进入 D2、真实 Key 或受控试运行前必须处理

1. 项目目录 ACL 继承范围过宽，其他本机用户可能修改代码、脚本或运行文件。
2. Caddy 本地 CA 私钥目录位于同一宽 ACL 范围。

处置要求：收紧项目运行目录与 CA 私钥目录 ACL，或由负责人针对隔离单用户环境作书面风险接受。该风险不阻断本机单用户脱敏 S1 Demo，但阻断真实 Provider Key 注入和试运行结论。

### P1：后续部署或多用户环境前处理

1. Caddy admin 仅监听 loopback，但没有独立认证。
2. PostgreSQL 当前网络监听状态不能等同生产隔离；本轮未修改数据库服务或 ACL。
3. 本地文件存储没有 NAS/对象存储级别的跨机备份、配额、生命周期和灾备。
4. Word 导出采用数据库与文件系统可恢复 Saga，不是跨资源原子事务；需要巡检 reserved/failed 版本。
5. 当前演示策略和条款不是校方法定采购口径。

### P2：建议改进

1. 在未来开放模板上传前加入 DOCX 主动内容与外部关系检查。
2. 增加 Word 视觉回归、LibreOffice/不同 Office 版本兼容性验证。
3. 增加高并发 S1 导出和磁盘耗尽故障演练。
4. 升级 Starlette/httpx 测试组合以消除现有弃用 warning，需独立依赖变更评估。

## 5. 不在本轮范围

- S3 库存。
- D2 在线 OCR/AI。
- Azure、DeepSeek、Redis、网关、NAS、MinIO。
- 生产部署、备份、监控、告警、灾备和容量测试。
- 真实业务模板、真实学校或供应商资料。
- PDF、邮件、打印和电子签章。

## 6. 回滚原则

- 应用层回滚只停用 S1 路由/导航或回退应用版本，不删除 PostgreSQL 数据和历史 Word。
- Migration 如需降级，必须先备份并确认没有需保留的 S1 数据；不得自动删除真实历史。
- 导出失败保留 failed 版本记录和不可复用版本号，通过受控 reconcile 处理。
- 不通过关闭 HTTPS、Secure Cookie、CSRF、Origin、RBAC 或审计来“恢复可用”。

## 7. 最终安全结论

S1 在本机、loopback、独立 PostgreSQL、脱敏数据和人工审批边界下可完成 Demo 验收。

当前没有阻断 S1 本机脱敏 Demo 的安全问题；P0 ACL 风险会阻断真实 Key、D2、受控试运行及生产化。S1 通过不代表生产可用或可受控试运行。
