# Steven 全项目风险与 D2 准入清单 v1

- 日期：2026-07-17
- 当前阶段：S1/S2/S3 本机独立脱敏 Demo 整体验收
- D2 状态：**未批准、未执行**
- 结论：当前风险不否定本机脱敏 Demo 验收，但阻断生产化、受控试运行和在线 Provider 接入。

## 1. 当前安全边界

- 浏览器唯一入口：`https://localhost:15443`
- Caddy、Next.js、FastAPI 与 Caddy admin 均仅监听 loopback。
- PostgreSQL 使用独立 Demo 数据库与最低权限应用账号。
- Session、CSRF、Origin/Referer、RBAC、审计和 legacy 身份头拒绝保持启用。
- OCR/AI live 开关关闭。
- 当前数据、文件和模板均为脱敏 Demo。

## 2. 风险清单

### 2.1 必须在任何真实 Key 或在线 Provider 联调前关闭

1. **项目目录与 TLS 私钥 ACL 过宽**
   - 当前项目目录及项目级 Caddy CA 私钥目录继承过宽权限。
   - 风险：其他本机用户可能修改运行代码或读取开发 CA 私钥。
   - Gate：收紧 ACL，或由负责人对隔离单用户测试机作书面风险接受。

2. **通用文件／任务／候选对象级授权需强化**
   - 通用 `files`、jobs、candidate 接口需在 D2 前明确 module、purpose、owner 和 scoped permission 的对象级授权矩阵。
   - 风险：未来多模块在线接入后可能发生横向读取。

### 2.2 进入受控联调前必须明确

1. **Caddy Admin 管理面**
   - 当前仅 loopback，但无认证。
   - Gate：明确禁用、受控 socket、访问控制或隔离环境风险接受方案。

2. **PostgreSQL 实例网络监听**
   - 当前数据库 ACL 与 `pg_hba.conf` 不允许本轮修改。
   - Gate：D2/部署环境需独立确认监听范围、网络边界和访问日志；不能把现状等同于生产隔离。

3. **原始文件名响应头**
   - 下载文件名需在未来真实资料接入前完成安全归一、字符过滤和敏感信息策略。

4. **request_id 输入限制**
   - 客户端传入 request_id 应增加长度、格式和来源约束，防止日志污染或超长输入。

5. **受控代理头信任**
   - Next.js 仅应信任受控 Caddy 传入的 `X-Forwarded-Proto`；未来拓扑变化时必须配置可信代理边界。

### 2.3 可在后续生产化阶段处理

- 多机部署、容量、压测、灾备、备份恢复、监控与告警。
- 文件存储从本地受控路径迁移到批准的存储服务。
- 统一审计检索、归档、保留期和不可抵赖性。
- Windows 服务化、证书生命周期和运维交接。

## 3. D2.0 负责人逐项决策

D2 计划不等于 D2 执行授权。负责人至少需要逐项书面决定：

1. Azure 区域、Endpoint、模型和 API 版本。
2. DeepSeek Endpoint、模型和严格 JSON Schema 支持方式。
3. 网络出口、代理、DNS、防火墙和出口日志责任边界。
4. 数据传输、保留、不训练、第三方日志及跨境/地域处理许可。
5. 可外发脱敏字段白名单与绝对禁止字段。
6. Key 托管、服务端注入、轮换、撤销和紧急停用机制。
7. Provider 失败 fallback：默认人工录入或人工重试；禁止把 Mock/缓存伪装为实时成功。
8. 项目目录、运行目录与 Caddy CA 私钥 ACL 收紧方案，或书面风险接受。
9. Caddy Admin 管理面的限制方案。
10. D2 停止条件、责任人、验收人、数据负责人、安全负责人和审计策略。

## 4. D2.1 Azure 最小联通 Gate

允许前提：

- D2.0 十项决策全部批准。
- 使用全新、最小权限、可撤销的服务端凭据。
- 只使用明确批准的脱敏样本。
- 固定区域、Endpoint、模型、API 版本和 timeout。
- live 与 Mock 结果在 UI、日志和审计中明确区分。

验收证据：

- 配置校验通过且日志不输出 Key。
- 单个脱敏样本真实请求成功。
- Provider request ID、模型、耗时、状态和失败原因可审计。
- 原始响应只按批准保留策略处理。

停止条件：

- 数据地域、保留或日志行为与批准不一致。
- 发现禁止字段外发。
- Key、文件内容或完整响应泄漏。

## 5. D2.2 DeepSeek 最小联通 Gate

允许前提：

- 固定 Endpoint、模型、Schema 版本和 timeout。
- 输入经过服务端脱敏构造。
- 输出必须经过严格 JSON Schema/Pydantic 校验。
- 输出只进入 `review_candidates(needs_review)`。

禁止：

- 直写 S1/S2/S3 正式表。
- 自动推荐供应商、计算可信金额、排序、审批、下单、发信或确认库存。
- 在 Schema 失败后宽松解析或默默补值。

## 6. D2.3 回退、人工复核与审计 Gate

- 网络失败、超时、限流、Schema 错误均进入明确失败状态。
- 默认回退为人工录入或人工重试。
- 缓存/Mock 只能显著标记，不得伪装为实时结果。
- 所有候选必须人工复核后由模块 confirmer + Unit of Work 写正式表。
- 审计关联 request_id、provider、model、文件、job、candidate、reviewer 和业务对象。
