# Steven S3 安全边界与遗留风险 v1

- 日期：2026-07-17
- 适用范围：本机独立脱敏 Demo
- 当前结论：没有阻断 S3 本机 Demo 验收的问题；存在阻断真实 Key、D2、受控试运行和生产化的安全前置项。

## 1. 已保持的安全边界

- 浏览器只通过 `https://localhost:15443` 访问；Caddy、Next.js、FastAPI 只监听 loopback。
- Session Cookie 保持 `Secure`、`HttpOnly`、`SameSite=Lax`、`Path=/`。
- CSRF Cookie 保持 Secure，所有状态变更继续要求合法 CSRF Token 与 Origin/Referer。
- S3 沿用 permission-based RBAC，不信任前端提交的 actor、角色、审批状态、差异值、建议订货量、版本号、文件路径或审计字段。
- 四种 legacy 身份头继续由 FastAPI 与 Next.js proxy 拒绝。
- 提交人不得审批自己的盘点单。
- staging/production 的 session 模式不得退回 InMemory S3 Repository 或内存审计。
- 正式 Excel 仅允许已批准盘点单生成；历史版本不得覆盖、删除或复用版本号。
- Next.js 子进程不继承 `NODE_TLS_REJECT_UNAUTHORIZED=0`。

## 2. 库存与 Excel 安全

已落实：

- SKU、品项、类别和库位均使用脱敏演示值。
- 数量字段使用严格非负整数校验；差异、低库存和建议订货量由服务端计算。
- 人工确认补货量偏离系统建议时必须提供人工理由。
- Excel 单元格使用公式注入防护，用户文本不能作为公式执行。
- 文件名与 storage key 由服务端生成，客户端不能指定正式路径或版本号。
- Excel 生成后重新打开校验并计算 hash/size。
- 本轮不支持任意盘点文件上传、宏文件、外部链接抓取、PDF、邮件、打印或外部存储。

需要明确的业务边界：

- 本模块记录账面快照、人工盘点值和补货建议，不确认实物库存真实性。
- 批准盘点单当前不会自动回写库存品项的账面数量；后续是否形成正式库存过账需校方确认并单独设计。
- 系统建议订货量是演示规则 `max(0, 目标库存 - 本次盘点量)`，不是自动采购决定。

## 3. OCR/AI 边界

- S3 OCR purpose 仅预留 `inventory_sheet_extraction`。
- S3 AI purpose 仅预留 `inventory_exception_explanation`。
- OCR/AI 输出只能进入 `review_candidates(needs_review)`。
- 人工确认前不得写 `steven_inventory_*` 正式表。
- AI 只能解释差异、低库存或字段异常。
- AI 不得确认实物库存、自动确认补货、自动下单、审批或修改正式库存值。
- 本轮没有调用 Azure、DeepSeek 或其他 Provider，没有读取或注入真实 Key。

## 4. 风险分级

### P0：进入 D2、真实 Key 或受控试运行前必须处理

1. 项目目录 ACL 继承范围过宽，其他本机用户可能修改代码、脚本或运行文件。
2. Caddy 本地 CA 私钥目录位于同一宽 ACL 范围。

处置要求：收紧项目运行目录与 CA 私钥目录 ACL，或由负责人针对隔离单用户环境作书面风险接受。该风险不阻断本机单用户脱敏 S3 Demo，但阻断真实 Provider Key 注入和试运行结论。

### P1：后续部署或多用户环境前处理

1. Caddy admin 仅监听 loopback，但没有独立认证。
2. PostgreSQL 当前网络监听状态不能等同生产隔离；本轮未修改数据库服务或 ACL。
3. 本地文件存储没有 NAS/对象存储级别的跨机备份、配额、生命周期和灾备。
4. Excel 导出采用数据库与文件系统可恢复 Saga，不是跨资源原子事务；需要巡检 reserved/failed 版本。
5. 当前未进行多实例、高并发盘点或磁盘故障压力验证。
6. 库存过账、单位换算、批次/有效期、调拨、报废和采购单联动均未定义。
7. 安全库存、目标库存和审批策略是脱敏 Demo 配置，不是校方正式制度。

### P2：建议改进

1. 后续开放盘点文件上传前加入文件大小、MIME/扩展名、恶意文件和压缩炸弹检查。
2. 增加 Excel 在不同 Office/LibreOffice 版本的兼容性验证。
3. 增加 S3 并发导出、磁盘耗尽和发布后数据库登记失败的故障演练。
4. 增加库存历史过账模型和不可变库存流水，但须先取得业务口径。
5. 升级 Starlette/httpx 测试组合以消除现有弃用 warning，需独立依赖变更评估。

## 5. 不在本轮范围

- D2 在线 OCR/AI。
- Azure、DeepSeek、Redis、网关、NAS、MinIO。
- 自动确认实物库存、自动补货、下单、邮件或供应商联动。
- 生产部署、备份、监控、告警、灾备和容量测试。
- 真实学校、物资、库位、供应商或库存资料。
- Word、PDF、打印和电子签章。

## 6. 回滚原则

- 应用层回滚只停用 S3 路由/导航或回退应用版本，不删除 PostgreSQL 数据和历史 Excel。
- Migration 如需降级，必须先备份并确认没有需保留的 S3 数据；不得自动删除真实历史。
- 导出失败保留 failed 版本记录和不可复用版本号，通过受控 reconcile 处理。
- 不通过关闭 HTTPS、Secure Cookie、CSRF、Origin、RBAC 或审计来“恢复可用”。
- 不通过自动覆盖账面数量或删除盘点单来清理演示数据。

## 7. 最终安全结论

S3 在本机、loopback、独立 PostgreSQL、脱敏数据和人工审批边界下可完成 Demo 验收。

当前没有阻断 S3 本机脱敏 Demo 的安全问题；P0 ACL 风险会阻断真实 Key、D2、受控试运行及生产化。S3 通过不代表生产可用或可受控试运行。
