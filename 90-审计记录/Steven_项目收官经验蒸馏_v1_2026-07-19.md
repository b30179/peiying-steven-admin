## 3. 最重要的架构决策

### 3.1 先冻结共用底座，再逐模块闭环

- Decision：先完成认证、Session、CSRF、RBAC、审计、PostgreSQL、HTTPS、文件和候选合同，再按 S2 → S1 → S3 完整闭环。
- Rationale：业务模块只实现领域模型和状态机，避免复制第二套认证、审计、文件、审批或 OCR／AI 基础设施。
- Result：三个模块可独立迭代，同时共享统一安全边界和运行时健康检查。

### 3.2 PostgreSQL 是唯一正式 Demo 持久化

- Decision：真实运行时必须报告各模块 store 为 PostgreSQL，禁止静默退回 InMemory。
- Rationale：单元测试通过不能证明重启恢复、审批历史、审计和文件版本真实持久化。
- Result：数据库模式、Session、审计、业务数据和版本元数据都能在重启后恢复。

### 3.3 AI／OCR 结果不直接写正式业务表

- Decision：OCR、DeepSeek 和规则提取先生成 `needs_review` 候选，再由人工确认写入业务数据。
- Rationale：识别和生成模型存在误读、漏项、格式漂移和错误推荐风险。
- Result：系统规则、AI 建议、人工决策、审批责任可以明确分离并审计。

### 3.4 导出采用追加式版本 Saga

- Decision：先预留版本号，再临时生成、可读性校验、计算 hash／size、发布并标记 ready；失败版本号不回收。
- Rationale：直接覆盖文件无法追溯，生成中断会留下数据库和文件状态不一致。
- Result：S1 Word 与 S2／S3 Excel 历史版本不可覆盖，可验证且可恢复。

### 3.5 浏览器只有 HTTPS 同源入口

- Decision：浏览器只访问 Caddy HTTPS，前端通过 Next.js API proxy 调 FastAPI；不新增弱 HTTP 登录或业务旁路。
- Rationale：防止绕开 Secure Cookie、CSRF、Origin／Referer 和 legacy 身份头拒绝。
- Result：认证与业务请求保持同一安全模型。

### 3.6 便携包与开发机状态分离

- Decision：发行 ZIP 不携带已初始化数据库集群、Session、日志、临时证书、运行状态、DPAPI 密钥或本机 AI 配置。
- Rationale：直接压缩已运行目录会把机器绑定状态和敏感材料带到目标电脑，也会破坏可重复初始化。
- Result：目标电脑从脱敏备份和运行时重新初始化，AI Key 在目标机单独配置并本机加密。

## 4. 成功经验

### 4.1 “一个业务模块一次完整闭环”比横向铺功能更可靠

每个模块都按数据模型 → Repository／UoW → API → UI → 审批 → 导出 → 审计 → PostgreSQL → HTTPS → 重启恢复完成，减少“页面存在但业务不闭环”的半成品。

### 4.2 验收必须区分三类证据

1. 真实运行时证据：PostgreSQL、HTTPS、Session、文件存储、重启恢复。
2. 自动化证据：pytest、compileall、ESLint、Next build、migration 检查。
3. 人工浏览器证据：按钮、布局、角色切换、错误提示、下载文件和操作顺序。

任何一类都不能代替另外两类。大量问题只会在真实页面和真实样本中出现。

### 4.3 负向规则应成为一等验收对象

项目价值不仅来自“能创建”，更来自明确阻断：自审、非法日期、预算错误、重复供应商／SKU、未替换变量、无理由偏离建议、未批准导出、退回无意见和重复版本覆盖。

### 4.4 Demo 数据必须服务于演示剧本

标准脱敏数据要主动覆盖成功与失败案例，而不只是填满表：

- S2 保留可解释的三供应商、五品项和固定金额结果。
- S1 保留多模板、多供应商、未替换变量和已批准版本历史。
- S3 保留至少 20 个品项、多库位、正负盘点差异、低库存和零／非零订货建议。

### 4.5 重置不是“清库”，而是受控业务操作

安全 reset 应具备 dry-run、显式双确认、目标清单、Demo 命名空间、事务／Saga、并发锁、失败记录、重新 seed 和 reset 审计；不得删除 schema、migration、账号、权限或非 Demo 文件。

### 4.6 AI 功能的产品价值来自“可操作”，不是“有结果”

AI 校对建议如果只能显示而不能定位和应用，就不算完成。最终交互需要：

- 明确原文、建议、位置和严重程度；
- 接受建议后真实修改当前草稿；
- 忽略也有状态；
- 修改后仍走业务规则和人工审批；
- 失败或超时明确显示 failed，不伪装 Mock 成功。

### 4.7 便携交付必须进行独立发行验证

开发目录能运行不等于交付包能运行。应在干净目录验证：解压、初始化、重复初始化、启动、重复启动、HTTPS、数据库、OCR 模型、停止、重复停止、状态丢失恢复和全文件哈希。

## 5. 踩坑与用户纠正

### 5.1 UI 布局不能只靠静态代码审查

出现过详情页追溯区域遮挡其他模块内容、盘点明细显示不全、实时预览挤压模板选择框、双栏内容过长和按钮样式不统一。经验是：复杂表单必须在真实数据量、真实窗口宽度和长文本下检查 overflow、sticky、min-width、滚动区和响应式断点。

### 5.2 用户说“扫描新增”，不能实现成“给已有记录补字段”

S1／S2 扫描入口最初放在详情页，和用户预期的新增流程不一致。产品语义应先明确：扫描是创建新事项、给现有事项追加资料，还是两者都支持；入口位置和后端 API 应与语义一致。

### 5.3 识别成功不等于业务提取成功

OCR 能读到字，但可能漏供应商、把“文件夹”识别成“文件央”、只提取首个供应商或把多供应商报价合成一份。测试必须使用真实版式样本，并分别验证文字层、版面分组、供应商边界和业务字段映射。

### 5.4 人工复核不能要求普通用户编辑 JSON

让首次使用者填写 `valid_until` 或修正候选时直接编辑 JSON，虽然技术上可行，但不具备可用性。应提供字段表单、行级异常、高亮证据、默认值说明和明确确认按钮；JSON 仅作为高级查看。

### 5.5 i18n 不能只修一批明显文案

繁简切换出现过大量固定中文遗漏。应集中维护消息键，并扫描组件中的用户可见字符串；业务数据、用户输入、SKU、文件名和导出内容不得被自动转换。

### 5.6 前端 proxy 方法必须和后端路由完整对齐

曾因 Next.js catch-all proxy 未导出 PUT，后端模板编辑 API 存在但浏览器返回 405。新增后端方法时必须同步检查 proxy、adapter、类型、CSRF 和前端调用。

### 5.7 “全部标为已读”需要定义可见结果

仅修改 unread 状态但通知仍原样显示，会让用户认为按钮无效。动作后应立即刷新状态、清除红点，并在列表中用已读样式、筛选或提示明确反馈；“已读”不等于“删除”。

### 5.8 角色名称和账号名称不能混用

用户明确纠正：Steven 是演示用户／产品命名，不应被当成抽象身份类型。最终演示只保留业务操作用户和独立审批用户，权限仍按 permission-based RBAC 实现。

### 5.9 Excel 导入要识别自己导出的工作簿

智能列映射曾被空列名、重复列名、标题行、合并单元格或元数据行阻断。导入器应先探测真正表头，忽略说明行和完全空列，规范化重复列名，并允许人工调整映射。

### 5.10 日期显示必须面向人而不是暴露序列化格式

Excel 曾直接输出带时区的 ISO 时间。存储层可以保留精确时间，但用户文件应统一为明确的本地日期时间格式，避免展示微秒和 `+08:00` 等实现细节。

### 5.11 启停脚本必须处理 Windows 中文路径和异步启动

曾出现 Caddy 日志路径乱码、Next.js 4300 等待超时和 PostgreSQL 便携版要求 ASCII 路径。脚本经验：

- CMD 使用 ASCII 内容、CRLF、无 BOM，保留真实退出码；
- PowerShell 统一 UTF-8 处理中文路径；
- 服务启动后检查进程是否提前退出，并保留 stdout／stderr 诊断；
- 等待健康端点而不是固定 sleep；
- 对 ASCII 路径要求提供自动工作目录／映射方案或清晰错误；
- start／stop／initialize 都必须幂等。

### 5.12 文档口径必须随阶段批准同步更新

项目早期 D2 未批准，后期负责人批准 DeepSeek 与 PaddleOCR 接入。若只追加文档而不清理旧表述，会同时出现“未调用 DeepSeek”和“DeepSeek 已验收”的冲突。阶段变化后必须全局检查 README、验收、风险、手册和交付说明。

### 5.13 真实密钥不能因为“方便演示”进入交付包

即使负责人要求提供可直接测试的 API，发行包仍不应包含真实 Key。正确做法是提供明确配置入口、服务端运行时读取、目标机本地加密、连通测试和关闭开关。

## 6. 可复用实施模板

### 6.1 新业务模块闭环模板

1. 明确状态机、角色、负向规则和导出条件。
2. 从当前 Alembic head 追加 migration，不修改历史 revision。
3. 复用认证、审计、files、jobs、candidates、版本化和 UoW。
4. 建 Repository Protocol 与 PostgreSQL 实现，禁止运行时 InMemory 回退。
5. API 禁止接受可信 actor、审批状态、审计字段和版本号。
6. UI 完成列表、创建、详情、修订、提交、审批、版本历史。
7. 用脱敏 seed 覆盖正常、边界和失败案例。
8. 验证 pytest、compileall、lint、build、migration、HTTPS 和重启恢复。
9. 写验收、实施、安全边界和演示脚本。

### 6.2 OCR／AI 接入模板

1. 上传文件进入通用 files，限制类型、大小和来源。
2. OCR job 只产出文字与 evidence，不写业务正式表。
3. AI job 使用 purpose 专用 Prompt 和严格 JSON Schema。
4. 禁止字段在 Schema、Prompt 和服务端再次校验。
5. 输出进入 `review_candidates(needs_review)`。
6. 前端展示证据、字段表单和行级异常，不强迫用户编辑 JSON。
7. confirmer 在 Unit of Work 内将人工确认结果写入业务表。
8. 失败、超时和关闭开关都有明确状态，不返回伪成功。
9. 日志记录 job、provider、耗时和错误类别，不记录 Key 或完整敏感原文。

### 6.3 便携交付模板

1. 构建发行 staging 目录，不直接打包开发工作区状态。
2. 包含锁定运行时、应用、migration、脱敏备份、OCR 模型和操作手册。
3. 排除 venv 缓存、数据库已初始化集群、Session、日志、临时证书、DPAPI 密钥和 AI 配置。
4. 初始化时检查路径、端口、运行时完整性和目标机前置条件。
5. Key 由目标机单独输入并本地保护。
6. 用临时 ZIP 构建，完整校验后原子替换正式文件。
7. 生成外部 `.sha256` 和包内全文件 `SHA256SUMS.txt`。
8. 在独立目录完成全生命周期 smoke 后再交付。
