## P0：必须在 Phase 1 方案和工程骨架中补齐

### P0-1 统一平台基础能力被弱化

`design.md` 要求底层共用登录/权限、文件上传、OCR/AI 任务、通知、审批、审计、导入导出和设置。此前任务书只写了 Steven 专属业务表及少量审批/审计关联，缺少明确的共用模块接入边界。

**修正**：工程骨架须包含或明确适配以下共用模块：`accounts`、`files`、`tasks`、`approvals`、`notifications`、`audit`。Steven 不实现其他岗位业务，但必须通过共用模块使用文件、审批、通知与审计。

### P0-2 异步任务技术栈不完整

设计要求是 **Celery / RQ + Redis**；此前文档仅写“Python Worker”。这不足以让 Codex 选定实现方式，也会造成任务状态不统一。

**修正**：采用 `Celery + Redis` 或 `RQ + Redis` 其中一种（由 Codex 基于现有项目确定，不能两套并存）。文档导出、批量 Excel 校验、未来 AI/OCR 统一走任务基础设施；任务状态采用 `pending / running / succeeded / failed / needs_review / confirmed`。Steven MVP 可不启用 OCR/LLM，但任务模型与可关闭 Adapter 必须预留。

### P0-3 文件存储与上传接口缺失

设计规定共享 `files` 表、统一上传能力和受控文件存储（NAS / MinIO / S3 兼容对象存储）；此前只说“校方受控目录”和 Steven 附件，未定义共用文件 API、文件实体、`source_file_id` 或存储适配层。

**修正**：提供 `/api/v1/files/upload`（并按权限限制下载/预览）；Steven 的模板、参考附件、报价附件、导出文档均关联 `files.id`，敏感业务记录带 `source_file_id`。文件存储用抽象接口，开发环境可本地目录，生产可切 NAS/MinIO/S3；禁止业务表直接存绝对路径。

### P0-4 API 合同未遵守统一响应规范

`design.md` 规定成功、列表、错误的固定 envelope（含 `request_id`、`pagination`、结构化错误）。此前任务书只列端点，没有返回格式、分页或错误码要求。

**修正**：全部 Steven API 使用：
- 成功：`{data, meta:{request_id}}`
- 列表：`{data, pagination:{page,page_size,total}, meta:{request_id}}`
- 错误：`{error:{code,message,details}, meta:{request_id}}`
并为列表端点落实分页、过滤、排序、空/加载/错误状态。

### P0-5 审批与通知只写状态，未形成平台流程

此前有 `pending_approval`，但缺少审批动作接口、审批人授权、通知创建和审计事件。

**修正**：通过共用 approvals 模块建立提交/批准/退回动作；提交审批、批准/退回、导出、删除、确认 AI 结果均写 `audit_logs`；待审批事件创建 `notifications` 记录。普通 Steven 不能自行批准本人任务。

## P1：应在 Phase 1/后续模块中落实

### P1-1 后端模块目录与测试边界

design.md 规定每个模块的 `router.py / schemas.py / models.py / service.py / repository.py / permissions.py / tests/` 分层。此前任务书只列接口与表，未约束分层。

**修正**：Steven 模块按该结构实现；Pydantic schema、SQLAlchemy model、服务层和 repository 不混写。测试至少覆盖 service 校验、API 权限、任务状态与导出版本规则。

### P1-2 统一 UI 组件和顶部栏遗漏

此前方案已有侧边栏、状态 Badge、抽屉等，但遗漏 design.md 指定的顶部“学校名称、当前用户、通知、退出”，并没有明确复用统一组件清单。

**修正**：Steven 页面必须复用/实现 `PageHeader`、`StatsCard`、`DataTable`、`FilterBar`、`UploadDropzone`、`TaskStatusBadge`、`AiReviewPanel`、`ConfirmDialog`、`AuditTimeline`、`FormSection`、`EmptyState`；顶部栏须有学校名称、当前用户、通知、退出。

### P1-3 AI Prompt 与结构化输出约束未落文

MVP 可不启用 AI，但一旦启用，prompt 不能散落在业务代码，输出必须为 JSON，且须有置信度、warnings、raw_text。

**修正**：预留 `apps/api/app/modules/steven/prompts/`；每个 prompt 写明输入字段、JSON schema、安全限制和示例。AI Adapter 默认 disabled，且 AI 输出先进入 `needs_review`，不直接落正式业务记录。

### P1-4 通用数据模型与敏感字段不足

design.md 的共用基础表包括 `users / roles / user_roles / permissions / role_permissions / modules / files / ocr_jobs / ai_jobs / tasks / approvals / notifications / audit_logs / system_settings`。此前未列 Steven 如何引用这些表，且需要追溯的业务记录未明确 `last_reviewed_by / last_reviewed_at / source_file_id`。

**修正**：Steven 业务表外键引用共用实体；含导入/附件/复核的记录按需加入上述追溯字段。

### P1-5 Phase 1 顺序应从“页面优先”调整为“模型和 API 定义先行”

设计规范要求：先明确模块文档 → 数据模型 → API → mock 页面 → 交互确认 → 接 API → 接 AI/OCR → 权限/审计/测试。此前给 Codex 的文本把 Open Design 页面和工程骨架放得过前，容易形成先做 UI 再补数据模型的倒序。

**修正**：Phase 1 应先产出 `Steven 模块设计说明 + 数据模型 + OpenAPI/API 合同`，再调用 Open Design 生成 Dashboard 并接入 mock adapter。Open Design 仍是硬任务，但不是替代模型和 API 设计。
