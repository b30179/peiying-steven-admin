- `P0-2`：业务数据、审批、审计和版本元数据随进程重启丢失；并发导出依赖内存计数，不能保证唯一版本、文件不覆盖和事务一致性。

## 2. 总体最小架构

```text
Browser
  -> Next.js（只持有 HttpOnly Session Cookie，不持有角色）
  -> FastAPI Auth Middleware / CurrentPrincipal
  -> RBAC Permission Check
  -> Steven Service（保留现有业务规则）
  -> PostgreSQL Repository + Unit of Work
  -> PostgreSQL：业务、会话、审批、审计、文件版本元数据
  -> 受控文件存储：本地受控目录 / NAS / MinIO（待校方确认）
```

统一原则：

- 身份由后端会话解析产生，前端不得提交 `actor`、`role` 或权限列表。
- 所有授权判断在 FastAPI 后端执行；Next.js 仅负责菜单、路由体验和 401/403 展示。
- 一次关键业务动作中的业务状态、审批、审计和版本元数据必须处于同一数据库事务或明确的可恢复流程。
- 业务 Service 不直接依赖内存字典或 SQLAlchemy 查询细节，通过 Repository / Unit of Work 接口运行。
- S1/S2/S3 复用同一账户、RBAC、审计、审批、文件和版本底座，不复制岗位专用认证逻辑。

# 3. P0-1：真实认证与权限边界

## 3.1 JWT / Session 最小选型

### 选型结论

第一阶段生产化采用 **PostgreSQL 支持的服务端 Session + HttpOnly Cookie**，不把 JWT 或角色信息暴露给浏览器脚本。

| 方案 | 优点 | 风险/复杂度 | P0 决策 |
|---|---|---|---|
| 数据库 Session Cookie | 可立即注销、撤权、锁定账户；适合校内管理系统；实现和审计清晰 | 每次请求需查询/缓存 Session | **采用** |
| 浏览器保存 JWT | 无状态、适合多服务 | 撤权困难；前端存储易扩大泄露面；角色变更不能即时生效 | 不采用 |
| HttpOnly JWT + Refresh Session | 可扩展到多服务 | 密钥轮换、刷新令牌、黑名单和重放防护复杂 | 暂不实施 |
| 校方 SSO/OIDC/SAML | 统一账户生命周期 | 依赖校方身份源和对接资料 | 待确认后作为后续适配器 |

JWT 预留方式：未来如 Worker 或独立服务需要服务间认证，可增加短时、限 audience 的服务 JWT；不得复用浏览器 Session，也不得在本次 P0 自动启用。

## 3.2 最小认证流程

1. 用户通过 `/login` 提交用户名和密码，或未来跳转校方 SSO。
2. FastAPI 验证身份；成功后创建随机 256-bit Session Token。
3. 数据库只保存 Token 哈希，不保存明文 Token。
4. 浏览器仅接收 `__Host-puiying_session` Cookie：`HttpOnly`、`Secure`、`SameSite=Lax`、`Path=/`，不设置可被子域覆盖的 Domain。
5. 每次请求由后端读取 Cookie、查询有效 Session、用户状态、角色和权限，生成只在服务端存在的 `CurrentPrincipal`。
6. 注销、账户停用、密码重置、角色撤销时，后端立即撤销相关 Session。
7. 未认证返回 401；已认证但无权限返回 403；两者不得被前端改写成成功。

建议默认策略：

- 空闲超时：30 分钟。
- 绝对会话时长：8 小时。
- 登录成功后轮换 Session ID，防止 Session Fixation。
- 修改密码、账户停用、角色变化时撤销所有既有 Session。
- 登录失败记录安全审计；连续失败达到阈值后短时锁定或延迟。
- 所有生产 Cookie 只允许 HTTPS；未启用 HTTPS 时不得标记生产可用。

超时和锁定阈值需由校方确认，以上为最小建议值。

## 3.3 最小账户与 RBAC 数据模型计划

后续新 migration 才新增，**本轮不修改现有 migration**：

- `users`：`id/username/display_name/email/password_hash/status/failed_login_count/locked_until/last_login_at/created_at/updated_at`。
- `roles`：`id/code/name/status`。
- `permissions`：`id/code/name/module`。
- `user_roles`：`user_id/role_id/assigned_by/assigned_at`，复合唯一。
- `role_permissions`：`role_id/permission_id`，复合唯一。
- `auth_sessions`：`id/user_id/token_hash/expires_at/idle_expires_at/last_seen_at/revoked_at/ip_hash/user_agent_hash/created_at`。
- 可选 `auth_events`：登录成功/失败、注销、撤权、会话过期；也可复用统一 `audit_logs` 并使用安全事件类型。

密码方案：使用成熟的自适应密码哈希算法及项目批准的库；具体库选择在实施阶段根据现有依赖和安全评估确认，本轮不安装依赖。

- 提交人自审必须由数据库记录的用户 ID 判断，不依赖用户名、显示名称或前端字段。
- Steven 只允许访问 Steven 业务对象；后续多岗位接入时，每个业务对象必须有模块归属或组织范围校验。

## 3.5 前端与后端职责

### Next.js 前端

- 通过 `/api/auth/me` 获取后端返回的当前用户显示信息和已裁剪的 UI capability。
- 根据 capability 显示或隐藏按钮，但隐藏只用于体验，不构成授权。
- 不允许设置 `X-Role`、`X-Actor`、`user_id`、`permissions` 等身份头或请求字段。
- 收到 401 跳转登录；收到 403 显示无权限，不重试为其他角色。
- 不把 Session Token、密码、权限列表写入 `localStorage`、前端日志或 URL。
- Next.js proxy 只转发 Cookie、请求体和 request ID；不得生成或覆盖身份。

### FastAPI 后端

- 唯一可信身份来源是生产 Session 或批准的 SSO Adapter。
- 每个 API 依赖明确权限，而不是仅判断角色名称。
- 从 `CurrentPrincipal.user_id` 写入 `created_by/updated_by/actor/submitted_by/decided_by`。
- 对对象范围、状态机、自审、审批权限、审计读取范围再次检查。
- 对客户端传入 `X-Role` / `X-Actor`：生产模式明确忽略并建议返回 400/403 安全错误，避免误以为生效。
- 审计记录认证失败、越权、审批、导出和管理员变更。

## 3.6 本地 mock 与生产认证隔离

建议配置：

- `APP_ENV=development|test|staging|production`
- `AUTH_MODE=mock|session|sso`

强制规则：

1. `AUTH_MODE=mock` 只允许 `development` 和 `test`。
2. `APP_ENV=staging|production` 且 `AUTH_MODE=mock` 时应用启动失败，不允许降级运行。
3. 取消通过任意业务请求头模拟角色；开发 mock 改为测试 dependency override，或仅绑定 loopback 的 `/dev/login/{fixture}` 创建 mock Session Cookie。
4. `/dev/login` 在 production 路由表中不存在，不是仅靠前端隐藏。
5. 测试身份使用固定 UUID 和脱敏名称，并与生产用户表、生产数据库完全隔离。
6. CI 分别运行 mock 模式测试和 session 模式集成测试，避免只测开发捷径。

## 3.7 P0-1 测试清单

### 鉴权

- 无 Cookie 请求 Steven API 返回 401。
- 无效、过期、已撤销 Session 返回 401。
- 登录后 Cookie 为 HttpOnly/Secure/SameSite；Session ID 登录后轮换。
- 注销、停用账户、撤销角色后旧 Session 立即失效。
- production 启动时 `AUTH_MODE=mock` 必须失败。
- 伪造 `X-Role: admin`、`X-Actor` 不改变权限。

### 越权与对象范围

- Steven 可访问 Steven S2，但不能访问其他岗位 API/页面数据。
- 审批人只能读取审批所需字段和附件，不获得修改报价权限。
- 管理员可管理账户和读全量审计，但默认不能修改报价或审批。
- 仅前端隐藏按钮时，直接请求 API 仍返回 403。
- 事项级审计只返回当前事项；Steven 请求全量审计返回 403。

### 审批与职责分离

- Steven 提交后不能调用 approve/reject。
- 审批人不能审批自己提交的事项，即使同时具有 Steven 与审批人角色。
- 第二个审批人重复处理已关闭审批返回 409。
- 管理员未叠加审批角色时调用审批 API 返回 403。
- 权限变更后已有页面继续发请求时，后端按新权限拒绝。

# 4. P0-2：PostgreSQL 持久化、事务与并发

## 4.1 Repository 分层迁移方案

保持现有业务 Service 的规则计算和状态机，替换数据实现：

```text
StevenQuoteService
  -> QuoteRepository Protocol
  -> UnitOfWork Protocol
       |- InMemoryUnitOfWork：仅开发/单元测试
       `- PostgresUnitOfWork：staging/production
            -> SQLAlchemy Session
            -> PostgreSQL
```

最小接口职责：

- `QuoteRepository`：事项、品项、供应商、报价明细查询与写入；不提交事务。
- `ApprovalRepository`：创建、锁定、批准、退回审批。
- `AuditRepository`：事务内追加不可变审计记录。
- `FileVersionRepository`：版本号预留、文件元数据、状态转换。
- `ImportBatchRepository`：预检批次、问题清单、人工确认状态。
- `UnitOfWork`：统一 transaction begin/commit/rollback；Service 决定事务边界。

禁止模式：Repository 方法内部各自 commit、业务写入成功后再单独写审计、用进程内 `len(versions)+1` 生成版本号。

## 4.2 现有 migration 执行顺序

已确认链：

1. `20260716_0001`：建立 S2 基础报价层级与 `UNIQUE(quote_supplier_id, quote_item_id)`。
2. `20260716_0002`：增量补齐 S2 工作流、追溯字段、导入批次、审批和版本表。
3. 后续新建 `0003+`：账户、角色、权限、Session、通用审计/文件底座和生产并发所需字段；不得回写或修改已验收的 `0001/0002`。

## 4.3 migration 执行、验证、备份与回滚

### 首次空库

1. 由运维创建独立数据库和最低权限应用账户。
2. 先运行 `alembic upgrade --sql`，保存离线 SQL 和 checksum。
3. 在一次性 staging 空库执行 `alembic upgrade head`。
4. 检查 `alembic_version`、所有表/外键/唯一约束/检查约束/索引。
5. 运行 schema smoke、Repository 集成测试和 3×5 S2 回归。
6. staging 验收通过后，才允许排期生产执行。

### 已有数据库

- 执行前记录当前 revision、数据库版本、扩展和容量。
- 使用 PostgreSQL 原生备份生成可恢复备份；备份完成后必须做恢复演练或至少在隔离库验证可读。
- migration 前进入维护窗口，暂停业务写入和 Worker。
- 所有 migration 先在同版本 staging 数据副本验证耗时、锁范围和回滚脚本。
- 执行后验证 revision、表行数、约束、关键查询和业务 smoke。

### 回滚策略

- schema 变更失败且事务未提交：依赖 PostgreSQL DDL transaction 回滚。
- 已提交但应用未切换：使用经过 staging 验证的 `alembic downgrade`，仅限可逆变更。
- 涉及删除/不可逆数据转换：不以 downgrade 作为唯一恢复手段，使用数据库备份恢复或前向修复 migration。
- 文件导出和数据库版本元数据必须一并核对；数据库回滚不得静默删除已经产生的正式文件。
- 生产执行前必须形成“数据库备份点 + 文件存储快照/版本点 + 应用版本”三者对应记录。

建议最低备份指标待校方确认；规划建议：每日全备、持续 WAL/增量、RPO 不高于 24 小时、RTO 不高于 4 小时。正式指标不能由开发组自行假定。

## 4.4 关键写操作事务边界

### A. 报价导入确认

单一数据库事务：

1. `SELECT quote_job FOR UPDATE` 锁定采购事项。
2. 校验事项仍可编辑、导入批次属于该事项且未确认。
3. 再次校验预检摘要/hash，防止使用已变化的批次。
4. 批量插入品项、供应商、15 条报价；依赖数据库唯一和检查约束兜底。
5. 更新批次 `confirmed_at/confirmed_by/status` 和事项状态。
6. 同一事务写 `quote.import_confirm` 审计。
7. 任一行失败全部回滚，不允许部分写入。

并发策略：同一事项同时确认两个批次时，后获得锁者重新校验；若事项已非空或批次已确认，返回 409。

### B. 人工推荐

单一数据库事务：

1. 锁定事项行。
2. 从数据库读取当前完整报价与币种；重新计算比较结果，不信任客户端排名。
3. 验证推荐供应商属于当前事项。
4. 非最低价时验证理由和审批意见均非空。
5. 更新推荐字段和 `updated_by`。
6. 同一事务写 before/after 审计。

### C. 提交审批

单一数据库事务：

1. 锁定事项行。
2. 再次检查完整比较、人工推荐和非最低价双字段。
3. 确认没有未关闭审批；数据库增加“每事项最多一个 pending 审批”的约束或部分唯一索引。
4. 插入 approval，记录 `submitted_by = principal.user_id`。
5. 更新事项为 `pending_approval`。
6. 同一事务写审计。

### D. 批准 / 退回

单一数据库事务：

1. `SELECT approval FOR UPDATE` 并锁定关联事项。
2. 验证审批仍为 pending、当前用户有 approve 权限。
3. 验证 `submitted_by != principal.user_id`，禁止自审。
4. 更新 approval、事项状态、`decided_by/decided_at/opinion`。
5. 同一事务写审计。
6. 并发第二次处理返回 409，不覆盖第一次决定。

### E. Excel 版本化导出

导出涉及数据库和文件系统，采用“数据库预留版本 + 临时文件 + 原子发布 + 数据库完成”的可恢复流程：

**事务 1：预留**

1. 锁定事项，确认状态为 approved/exported、比较仍有效。
2. 原子递增事项的 `next_export_version`，或锁定单独的版本计数行。
3. 插入 `file_versions` 记录：`version_number/status=reserved/created_by`。
4. 提交事务。版本号一旦预留不重复使用；失败可留下 gap，但绝不复用覆盖。

**文件生成**

5. 使用包含 `quote_id + version_number + random UUID` 的临时路径生成 XLSX。
6. 重新打开验证工作簿、计算 SHA-256 和文件大小。
7. 使用对象存储 put-if-absent，或同一文件系统内原子 rename 发布到唯一 storage key；目标已存在时必须失败，禁止覆盖。

**事务 2：完成**

8. 锁定版本记录，将 `status=ready`，写入 `storage_key/sha256/size/mime/created_at`。
9. 同一事务写 `quote.export` 审计并更新事项最近导出信息。

失败恢复：

- 文件生成失败：版本记录标记 `failed`，保留失败原因和审计；版本号不回收。
- 文件已发布但数据库完成失败：后台恢复任务按 storage key/hash 对账并完成或隔离；不得创建同版本新文件。
- 数据库已完成但文件缺失：下载返回明确错误并触发告警，不得伪装成功。

## 4.5 并发版本号与历史文件保护

数据库约束：

- `UNIQUE(business_object_type, business_object_id, version_number)`，或 S2 现有 `UNIQUE(quote_job_id, version_number)`。
- 事项增加原子版本计数，避免 `MAX(version)+1` 在并发下竞争。
- `storage_key` 唯一，正式版本记录不可 update 为另一文件。
- 文件状态：`reserved -> ready | failed`，禁止 `ready -> reserved`。

文件命名仍为：

```text
YYYYMMDD_采购比价_事项简称_v{版本}.xlsx
```

实际 storage key 需包含稳定对象 ID，不能只依赖可重复的中文文件名：

```text
steven/quotes/{quote_id}/v{version}/{uuid}.xlsx
```

下载时使用元数据提供正式文件名；存储层始终 append-only。

## 4.6 必须持久化的字段

### 业务对象

- 所有 S2 事项、品项、供应商、报价明细。
- `id/status/created_at/updated_at/created_by/updated_by`。
- 推荐供应商、非最低价理由、审批意见、统一币种、有效期、运费、税费。
- 导入批次 filename/hash、解析计数、issues、状态、确认人和确认时间。

### 审批

- `approval_id/object_type/object_id/status/submitted_by/submitted_at/decided_by/decided_at/opinion`。
- 状态历史或足够的审计 before/after，确保重启后可追溯。

### 审计

- `id/actor_user_id/action/object_type/object_id/timestamp/request_id/ip_hash/before_after`。
- 审计 append-only；普通业务 API 不提供 delete/update。
- `before_after` 避免存放文件原文、密码、Session Token 或密钥。

### 文件与版本

- `id/object_type/object_id/version_number/status/original_filename/storage_key/mime_type/size/sha256/created_at/created_by/failure_reason`。
- 正式文件本体存储于受控文件系统/NAS/对象存储，数据库保存元数据与完整性 hash。

### 认证

- 用户、角色、权限、Session 撤销和到期信息必须重启后保留。

## 4.7 脱敏演示数据与真实数据隔离

最小策略：**独立数据库 + 独立文件根目录 + 独立凭据**。

- development/test 使用 `puiying_steven_dev/test` 和脱敏 seed。
- staging 可使用独立 staging 数据库，仅允许脱敏或批准的测试数据。
- production 使用独立数据库和文件 bucket/root；启动时默认不执行 demo seed。
- `SEED_DEMO_DATA=true` 在 production 必须导致启动失败。
- 不依赖单个 `is_demo` 字段隔离真实数据；该字段只能用于显示标识，不能替代环境隔离。
- 测试、截图和 CI 输出不得连接生产数据库或生产文件存储。
- 从生产复制到测试必须经过批准的脱敏导出流程，本阶段不实现。

## 4.8 P0-2 测试清单

### migration

- 空库按 `0001 -> 0002 -> 后续 0003+` 升级成功。
- 每一步 revision、外键、唯一约束、check、索引与 server default 验证。
- staging 数据库 downgrade/前向修复演练。
- 备份可恢复到隔离库，恢复后关键行数/hash 一致。
- migration 中断后应用不得在半完成 schema 上启动。

### Repository 与事务

- InMemory 与 PostgreSQL Repository 对同一 Service contract 产生一致结果。
- 导入第 15 行失败时前 14 行不落库，审计也不落半条。
- 推荐更新失败时业务字段与审计同时回滚。
- 审批创建失败时事项状态不变。
- 审批成功后重启 API，状态、审批人和审计仍存在。
- API 重启后 3×5 数据、推荐、审批、导出版本均可恢复。

### 并发

- 两个请求同时确认同一导入批次：一个成功，一个 409，无重复行。
- 两个请求同时写同一供应商+品项：唯一约束保证只保留一条。
- 两个审批人同时处理：仅一个成功，另一个 409。
- 10 个并发导出请求获得不重复版本号；历史文件均存在且 hash 不被改变。
- 导出生成中进程被终止：重启后 reserved/failed 记录可对账，不覆盖已发布文件。

### 隔离与回归

- production 启动不产生 demo 数据。
- dev/staging 凭据无法连接 production。
- Phase 1 Dashboard 与 S2 原有 17 项规则测试继续通过。
- 认证接入后跨角色、全量审计、自审测试继续通过且不再依赖身份头。

## 4.9 P0-2 实施分类

### 可立即实施的最小改造

- 定义 Repository / Unit of Work 接口和 PostgreSQL 实现边界。
- 新增后续 migration，而不是修改 `0001/0002`。
- 将导入、推荐、审批、审计写入纳入明确事务。
- 以数据库原子计数和唯一约束解决并发版本号。
- 建立 Session、审计、审批、文件版本的持久化表和重启恢复测试。
- 禁止 production 自动 seed，建立 dev/test/prod 数据库配置隔离。

### 需要校方确认的事项

- PostgreSQL 运行位置、运维责任方、维护窗口和允许停机时间。
- 数据库 RPO/RTO、备份频率、保留期和恢复演练频率。
- 正式文件使用校内 NAS、MinIO 或其他 S3 兼容存储；访问、备份和保留政策。
- 审计日志、审批记录和正式 Word/Excel 的法定/校内保留年限。

### 暂不实施的事项

- Redis/Celery 异步导出、跨区域数据库、读写分离、高可用集群。
- 多租户/RLS、事件溯源、CDC、数据仓库和全文审计分析。
- 自动清理正式文件、自动归档到外部系统。
- 真实生产数据迁移；需在 schema、权限和备份方案获批后另立迁移计划。

# 5. S1 / S2 / S3 共用底座复用

| 共用能力 | S1 标书/报价单 | S2 采购比价 | S3 消耗品库存 |
|---|---|---|---|
| Session / CurrentPrincipal | Steven 登录、提交人、审批人 | Steven、推荐人、审批人、导出人 | Steven、盘点录入人、人工确认人 |
| RBAC | `steven:tenders:*` | `steven:quotes:*` | `steven:inventory:*` |
| Unit of Work | 草稿生成、提交、审批、版本记录 | 导入、推荐、审批、导出 | 导入、盘点、差异确认、导出 |
| 审计 | 模板、字段、草稿、审批、导出 | 报价、推荐、审批、导出 | SKU、盘点差异、补货建议、确认 |
| Approval | 文书正式导出前审批 | 推荐后正式导出审批 | 如校方需要，可用于补货建议审批；不自动下单 |
| File / Version | Word Draft 与正式版本 | 比价 XLSX 版本 | 盘点/低库存 XLSX 版本 |
| 数据隔离 | Steven 模块对象范围 | Steven 模块对象范围 | Steven 模块对象范围 |
| AI Adapter | 未来仅草稿辅助，人工复核 | 未来只整理说明，不推荐供应商 | 未来只解释建议，不确认库存 |

复用方式：

- 平台共用 `accounts/auth/rbac/audit/approvals/files` 模块。
- 岗位模块只定义业务表、业务权限代码和状态机，不复制 Session 表或审计实现。
- 文件版本使用统一 `object_type/object_id/version_number` 合同，S1/S2/S3 只提供各自生成器。
- 审批以通用对象引用关联，不在每个岗位重复实现身份校验和自审检查。

# 6. 推荐实施顺序

虽然需求按 P0-1、P0-2 描述，但生产 Session 本身依赖数据库，因此实际实施顺序为：

1. **决策确认**：账户来源、审批角色、PostgreSQL/文件存储和备份指标。
2. **数据库基础**：配置隔离、连接/Session/Unit of Work、空库 migration 验证；不切换业务流量。
3. **认证与 RBAC**：用户/角色/权限/Session migration，替换身份头，完成 401/403/自审测试。
4. **S2 PostgreSQL Repository**：保持 Service 规则不变，完成导入/推荐/审批事务和重启恢复。
5. **版本与文件并发**：版本预留、原子发布、失败恢复和 10 路并发导出测试。
6. **全量回归**：Phase 1 Dashboard、S2 17 项规则、认证、数据库、备份恢复和浏览器 smoke。
7. **受控试运行评审**：只有校方确认基础设施与测试证据后，S2 才可从“开发验收完成”升级为“可申请试运行”。

# 7. 最小完成标准

- 生产模式完全不信任 `X-Role` / `X-Actor`，前端无法伪造身份。
- Session 可登录、过期、注销、撤销；用户和角色变更即时生效。
- Steven、审批人、管理员权限矩阵由后端测试证明。
- S2 所有业务、审批、审计和版本元数据在 API 重启后保留。
- 关键写操作事务原子；导入或审批失败不产生半完成记录。
- 并发导出不重复版本、不覆盖文件，可识别和恢复失败状态。
- migration、备份、恢复、回滚和生产配置隔离均有证据。
- Phase 1 和 S2 既有验收行为无回归。
- AI/OCR、自动选商、自动审批、自动下单、发信和库存确认仍保持禁用。
