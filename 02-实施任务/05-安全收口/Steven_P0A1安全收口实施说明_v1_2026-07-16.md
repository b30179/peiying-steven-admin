## 1. 首个管理员安全初始化

新增一次性命令：

```powershell
cd D:\ZM\AI工坊\某中学\04-开发实现\apps\api
$env:APP_ENV="development"
$env:AUTH_MODE="session"
$env:DEMO_SEED_ENABLED="false"
$env:DATABASE_URL="postgresql+psycopg://<user>:<password>@<host>:<port>/<database>"
$env:BOOTSTRAP_ADMIN_USERNAME="<initial_admin_username>"
$env:BOOTSTRAP_ADMIN_DISPLAY_NAME="<initial_admin_display_name>"
$env:BOOTSTRAP_ADMIN_PASSWORD="<read_from_managed_secret>"
..\..\.venv\Scripts\python.exe -m app.cli bootstrap-admin
```

安全约束：

1. migration 不创建管理员、不写默认账号或密码，应用启动也不会自动创建管理员。
2. 用户名、显示名和初始密码只从环境变量读取；密码至少 12 个字符，命令输出和错误信息不回显密码。
3. PostgreSQL 实现使用 transaction advisory lock 防止并发创建；已有管理员时返回 `admin_already_exists` 并拒绝重复 bootstrap。
4. 成功与拒绝均写入 `auth_security_events`，事件不包含密码。
5. 失败时先确认 migration 已到 `20260716_0004`、admin 角色存在和数据库连接权限；不得通过修改 migration 或删除管理员绕过保护。
6. 本轮不提供自动恢复或覆盖管理员命令。若首管理员失去访问能力，应走受控恢复变更，由两人复核数据库备份、身份与审计后另行实施。
7. 初始密码必须为一次性受管值；首次安全登录后应按校方密码策略轮换。本轮未实现生产密码轮换页面，因此在该能力和批准流程完成前不得进入生产。

`.env.example` 仅包含 `<...>` 占位符，没有真实凭据。

## 2. CSRF 与请求来源校验

Session 登录成功后，FastAPI 同时写入：

- `__Host-puiying_session`：随机 Session Token，仅 HttpOnly Cookie；数据库只存 SHA-256 hash。
- `__Host-puiying_csrf`：独立随机、前端可读、Secure、SameSite=Lax Cookie。

在 `AUTH_MODE=session` 下，所有 `POST/PUT/PATCH/DELETE` 统一经过中间件：

1. 严格校验 `Origin`；无 Origin 时只接受可解析且同源的 `Referer`。
2. 来源必须精确匹配 `ALLOWED_ORIGINS`，staging/production 只允许 HTTPS Origin。
3. 除登录外，必须同时存在 Session Cookie、CSRF Cookie 和 `X-CSRF-Token`，并以恒定时间比较 Cookie/Header Token。
4. 登录不要求登录前不存在的 CSRF Token，但仍强制 Origin/Referer 校验。
5. CSRF、Origin、旧身份头拒绝均写入认证安全事件，事件只记录错误码、方法和路径。

覆盖账户创建、角色分配、账户停用、Session 撤销、注销、采购导入预检/确认、人工推荐、提交审批、批准/退回、导出及后续新增的所有 Cookie Session 写接口。

Next.js 使用共享代理安全模块转发 Cookie、Origin、Referer、CSRF 和 Request ID；React 写请求通过 `securedFetch` 在存在 CSRF Cookie 时自动加入请求头。前端不能读取 Session Token，也不向 URL、日志或状态写入 Session Token。

## 3. 旧身份头统一拒绝

FastAPI middleware、`/api/v1/auth/login` 与 Next.js 两类 proxy 使用同一规则，任何环境均拒绝：

- `X-Role`
- `X-Actor`
- `X-Acting-Role`
- `X-Acting-Actor`

统一返回 HTTP 400 与错误码 `legacy_identity_headers_forbidden`。mock 身份只允许由 development/test 的服务端 `MOCK_IDENTITY` 或测试 dependency/repository 产生，业务请求头不能覆盖身份。

## 4. 登录防爆破与审计

- 账户级失败次数和短时锁定由 Auth Repository 维护；默认连续失败 5 次锁定 15 分钟。
- 成功登录清零 `failed_login_count/locked_until` 并更新 `last_login_at`。
- 所有密码只进入密码校验，不进入日志、API 错误、审计 details 或 CLI 输出。
- 新增无依赖、单进程、按客户端地址和用户名组合的短窗口限流；默认 5 分钟最多 10 次失败尝试，超限返回 429。
- 登录失败、限流、Session 缺失/无效、CSRF/Origin/旧身份头拒绝、权限拒绝、bootstrap、角色变更、Session 撤销和账号停用均有认证安全事件。
- 进程内限流不能在多实例之间共享；多实例生产部署前仍需由反向代理或共享限流存储补齐。这是明确安全边界，不宣称已完成分布式防爆破。
