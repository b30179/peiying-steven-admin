## 1. 实施结果

### 1.1 环境与启动保护

新增统一配置：

- `APP_ENV=development|test|staging|production`
- `AUTH_MODE=mock|session|sso`
- `DEMO_SEED_ENABLED`
- `DATABASE_URL`
- `FILE_STORAGE_ROOT`
- `SESSION_IDLE_MINUTES=30`
- `SESSION_ABSOLUTE_HOURS=8`
- `SESSION_COOKIE_NAME=__Host-puiying_session`
- `SESSION_COOKIE_SECURE=true`

应用启动时强制执行：

1. `staging/production + AUTH_MODE=mock` 启动失败。
2. `production + DEMO_SEED_ENABLED=true` 启动失败。
3. `staging/production + SESSION_COOKIE_SECURE=false` 启动失败。
4. `AUTH_MODE=sso` 在校方 SSO Adapter 未配置前启动失败。
5. `staging/production + AUTH_MODE=session` 必须提供非占位 `DATABASE_URL`。
6. development/test 的 mock 身份只由服务端 `MOCK_IDENTITY` 固定产生，不读取业务请求身份头。

development、test、staging、production 已提供独立数据库名、文件根目录和配置模板；所有密码、主机、NAS 路径和密钥均为占位符，没有写入真实校方信息。

### 1.2 服务端 Session

- 登录成功创建随机 Session Token；浏览器只持有 HttpOnly、Secure、SameSite=Lax、Path=/ Cookie。
- 数据库只保存 SHA-256 Token hash，不保存明文 Token。
- 默认空闲超时 30 分钟、绝对时长 8 小时；解析 Session 时同时检查用户状态、撤销、空闲到期和绝对到期。
- 注销、管理员撤销 Session、停用账户、变更角色后，旧 Session 立即撤销。
- 本地密码采用 Python 标准库 `hashlib.scrypt` 的带盐哈希格式；没有保存或提供真实账户密码。

### 1.3 Permission-based RBAC

FastAPI 现从 Cookie 或受限 mock 配置生成 `CurrentPrincipal`，业务端点通过权限 dependency 授权，不再仅按角色名称或客户端身份头判断。

| 能力 | Steven | 审批人 | 管理员 |
|---|---:|---:|---:|
| Dashboard、S2 读取 | 允许 | 仅审批所需读取 | 默认禁止 |
| S2 写入、导入、人工推荐、提交 | 允许 | 禁止 | 禁止 |
| S2 批准、退回 | 禁止 | 允许 | 默认禁止 |
| 已批准 S2 导出 | 允许 | 禁止 | 禁止 |
| 事项级审计 | 允许 | 允许 | 通过全量审计权限读取 |
| 全量审计 | 禁止 | 禁止 | 允许 |
| 账户、角色、会话管理 | 禁止 | 禁止 | 允许 |

管理员没有隐含审批权；只有显式授予审批角色才获得 `steven:quotes:approve`。同一用户即使同时具有 Steven 与审批角色，只要该用户是事项提交人，仍由服务端以稳定用户 ID 阻断自审。

### 1.4 前端身份边界

- Next.js 不再生成、传入或覆盖 `X-Role`、`X-Actor`、`X-Acting-*`。
- Next.js Steven proxy 只转发 Cookie 和业务请求；检测到身份覆盖头时返回 400。
- FastAPI 同样拒绝旧身份头，伪造 `X-Role: admin` 或 `X-Actor` 不会改变真实权限。
- 新增 `/login` 与 `/api/auth/[...path]`；Session Token 只由 FastAPI 写入 HttpOnly Cookie，不进入 React 状态、localStorage 或 URL。
- 前端按钮显示只用于体验，401/403 和所有业务授权仍由 FastAPI 决定。

## 2. PostgreSQL 基础底座

### 2.1 Migration 链

既有 migration 保持不变，只追加：

```text
20260716_0001
→ 20260716_0002
→ 20260716_0003
```

`20260716_0003_auth_rbac_sessions.py` 新增：

- `users`
- `roles`
- `permissions`
- `user_roles`
- `role_permissions`
- `auth_sessions`

schema 包含必要主键、复合唯一、外键、状态检查、到期检查和查询索引。`auth_sessions.token_hash` 有唯一约束；Session 具备 `idle_expires_at/absolute_expires_at/revoked_at/revoke_reason/is_active`。migration 只初始化固定角色和权限，不创建真实用户、密码或校方资料。

Alembic 运行时优先读取 `DATABASE_URL`，避免 migration 使用与应用环境不一致的连接入口。

### 2.2 Repository 状态

- 已提供 `AuthRepository` 合同、测试用 `InMemoryAuthRepository` 和 `PostgresAuthRepository`。
- session 模式会创建 PostgreSQL Engine，并将账户、角色和 Session 身份边界接到 PostgreSQL Repository。
- S2 业务数据仍使用已验收的内存 Repository；这是本轮边界，不代表 S2 已持久化。
