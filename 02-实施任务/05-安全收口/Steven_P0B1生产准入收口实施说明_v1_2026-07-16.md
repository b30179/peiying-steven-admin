## 1. Migration 链

既有 revision 未修改，仅追加 `20260716_0006`：

```text

```

`0006` 在创建索引前按 `lower(username)` 分组预检历史账号；发现冲突时以 PostgreSQL `23505` 中止，并在错误中报告冲突用户名集合。migration 不合并、不删除、不重命名账号。预检通过后创建：

```sql
CREATE UNIQUE INDEX uq_users_username_lower ON users (lower(username));
```

同时新增 `platform_audit_events` 及 request、object、actor、action 时间索引。离线 SQL 证据为 `05-测试验收/Steven_P0B1_offline_migration_0006.sql`。

## 2. 用户名唯一与冲突语义

1. PostgreSQL 以 `uq_users_username_lower` 保证数据库级大小写不敏感唯一，应用层查询不作为最终并发保证。
2. `PostgresAuthRepository.create_user` 与 `bootstrap_admin` 只把 `uq_users_username`、`uq_users_username_lower` 的 `IntegrityError` 映射为 `duplicate_username`；其他约束错误继续抛出。
3. 首管理员 bootstrap 保留 PostgreSQL advisory transaction lock，已有管理员仍返回 `admin_already_exists`；无管理员但用户名与已有普通账户大小写冲突时返回 `duplicate_username`。
4. development/test 内存 Repository 使用锁与 `casefold()` 维持同一业务语义，但不冒充数据库并发证明。
5. 空库 smoke 已编排两个管理员 Session 并发创建大小写变体用户名，预期一个 `201`、一个 `409 duplicate_username`；该步骤尚未在真实 PostgreSQL 执行。

## 3. 通用平台审计

### 3.1 持久化范围

`PostgresAuditRepository` 将平台事件写入 `platform_audit_events`。`PostgresAuthRepository` 的认证安全事件与对应平台审计在同一数据库事务写入，覆盖：

- 登录成功、登录失败、账户锁定、登录限流；
- Session 缺失、无效、过期或撤销后的认证拒绝；
- 权限拒绝；
- bootstrap 成功/拒绝；
- 账户创建、账户停用、角色变更及由此触发的 Session 撤销；
- 登录、登出和管理员主动撤销 Session。

HTTP middleware 将 `X-Request-Id` 或服务端生成 UUID 放入 `ContextVar`；平台审计与 S2 报价审计自动写入同一 `request_id`，响应也返回该 ID。不得在审计详情中写密码、Session Token、CSRF Token 或密钥。

### 3.2 查询、关联与保留

- 平台审计：管理员通过全量审计 API 查询，普通 Steven 用户无全量读取权限。
- S2 报价审计：Steven/审批人在事项权限边界内读取；写入继续与 S2 业务 UoW 同事务。
- 关联：优先按 `request_id` 关联一次 HTTP 请求的认证/授权事件与报价业务事件，再以 `object_type + object_id`、actor 和时间定位对象历史。
- 保留：两类审计均为追加式持久数据，普通用户不能删除；具体保留年限、归档介质和法务要求仍需校方确认。本轮不实现自动清理。
- 环境：内存审计只允许 development/test fixture；staging/production Session 模式显式拒绝内存平台审计。

## 4. 多实例限流与部署保护

提供统一 `LoginRateLimiter` 接口：

- `MemoryLoginRateLimiter`：仅 development/test；单进程，不视为生产防爆破完成。
- `RedisLoginRateLimiter`：定义共享后端的 allow/record/clear 契约与 fake 测试；当前不创建 Redis 客户端、不连接真实 Redis。
- `GatewayLoginRateLimiter`：由获批准的反向代理/WAF 在上游执行共享限流；应用保留账户失败计数和短时锁定。

staging/production 必须显式设置 `RATE_LIMIT_MODE=redis|gateway`；未配置或 `memory` 时启动失败。Redis 模式缺 `REDIS_RATE_LIMIT_URL` 时失败；gateway 模式缺 `TRUSTED_PROXY_CIDRS` 时失败。

客户端 IP 默认只取 socket peer。仅当 socket peer 位于 `TRUSTED_PROXY_CIDRS` 时，才解析 `X-Forwarded-For` 的第一个合法 IP；不盲信任任意转发头。当前 `.env.example` 仅有占位符，无真实服务地址或 CIDR。

## 5. 历史审批 actor 治理

`submitted_by`、`decided_by` 外键仍保持 `NOT VALID`。新增 `scripts/govern_s2_approval_actors.py`：

1. 默认只读报告孤儿 actor 与两个约束的 `convalidated` 状态。
2. 映射写入必须同时提供 `--apply-approved-mapping`、显式 JSON 映射文件和 `P0B1_ACTOR_GOVERNANCE_APPROVED=true`。
3. 映射目标必须是现存用户；工具不猜测、不删除、不自动合并历史数据。
4. 仅在重新报告 orphan 为零后，操作人员才可显式使用 `--validate` 执行 `VALIDATE CONSTRAINT`。

当前未连接 PostgreSQL，故没有历史孤儿报告，两个约束也未实际 VALIDATE。进入受控试运行前必须由校方批准映射/清理决定，并在可审计变更窗口执行。

## 6. 导出一致性与巡检

S2 导出明确属于“可恢复一致性 Saga”，不是数据库与文件系统的跨资源原子事务：

```text
数据库预留版本
→ 临时文件生成与校验
→ 不可覆盖发布
→ 数据库 ready 登记
→ 失败/中断由巡检与人工授权 reconcile 收口
```

新增 `scripts/scan_s2_reserved_exports.py` 只读扫描超时 `reserved`，输出 `ready_candidate`、`failed_candidate` 或 `manual_review`。它不更新数据库、不删除/覆盖文件、不回收或复用版本号。现有 reconcile 动作继续要求人工授权；历史文件和版本号始终追加保留。

## 7. PostgreSQL 空库验证入口

`scripts/verify_p0b_postgres.py` / `.ps1` 已更新，计划在获批准环境覆盖：

- `alembic upgrade head/current` 与 `0006` 关键索引、表、约束；
- bootstrap、登录、Session、CSRF、RBAC；
- 大小写用户名直接冲突与并发创建；
- 平台审计重启后读取及 `request_id` 关联；
- 3×5 导入、重启恢复、推荐、审批、自审、并发导入/审批；
- 10 路导出版本、文件 hash/size；
- actor 外键 `convalidated` 状态。

脚本还要求 `RATE_LIMIT_MODE=gateway` 与非空 `TRUSTED_PROXY_CIDRS`，从而验证部署保护配置；它不连接 Redis。缺必要配置时 exit 2，且在数据库连接前停止。
