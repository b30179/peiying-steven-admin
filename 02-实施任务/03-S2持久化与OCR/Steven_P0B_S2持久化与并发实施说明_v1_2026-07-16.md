## 1. 应用分层与环境边界

S2 路由通过 `request.app.state.quote_application` 调用 `StevenQuoteApplicationService`，业务规则继续由 `StevenQuoteService` 负责，路由、前端和 Repository 不复制金额计算、完整性阻断或状态机规则。

```text
FastAPI Router
  → StevenQuoteApplicationService
      → QuoteUnitOfWork
          ├─ InMemoryQuoteUnitOfWork（development/test fixture）
          └─ PostgresQuoteUnitOfWork（staging/production）
      → StevenQuoteService（既有业务规则）
      → PostgresQuoteRepository / PostgresQuoteAuditRepository
      → LocalAppendOnlyQuoteStorage（当前仅受控本地路径）
```

环境约束：

1. staging/production 创建 lazy PostgreSQL S2 application；首次使用时建立数据库 Engine，不回退内存业务数据。
2. development/test 可继续使用内存 Repository 作为 fixture，不代表持久化或 PostgreSQL 集成验证。
3. staging/production 继续要求 `AUTH_MODE=session`、独立 `DATABASE_URL`、独立 `FILE_STORAGE_ROOT`、HTTPS Origin，并禁止 demo seed。
4. 认证、permission-based RBAC、CSRF、Origin、旧身份头拒绝和自审阻断沿用 P0-A / P0-A.1 后端边界；前端不能提交角色或 actor 覆盖值。
5. 未连接 NAS、MinIO 或校内正式文件存储；当前 storage 实现是可替换边界下的本地追加式开发实现。

## 2. Migration 链与数据模型

既有 revision 未修改，仅追加：

```text
20260716_0001
→ 20260716_0002
→ 20260716_0003
→ 20260716_0004
→ 20260716_0005
```

`20260716_0005` 补齐：

- `steven_quote_jobs`：脱敏标记、当前审批关联、`next_export_version` 原子计数器及状态约束。
- `steven_quote_import_batches`：预检结果、规范化 payload、payload SHA-256、确认人/时间、状态、更新时间及事项/状态索引。
- `steven_quote_approvals`：提交/决定时间、审批状态约束、提交人/决定人关联，以及“每事项最多一个 pending 审批”的 PostgreSQL 部分唯一索引。
- `steven_quote_versions`：对象类型/ID、版本号、`reserved/ready/failed`、正式文件名、storage key、mime、size、SHA-256、失败原因、临时 key、发布时间及更新时间。
- `steven_quote_audit_events`：actor、action、对象、时间、before/after JSONB，并建立对象/actor 时间索引。
- 继续保留 `UNIQUE(quote_supplier_id, quote_item_id)`、事项内品项/供应商唯一约束、金额非负和数量正数约束。

兼容说明：历史审批 actor 字段原为自由文本。`submitted_by` / `decided_by` 外键以 `NOT VALID` 追加，避免升级时把历史非 UUID actor 误写为合法账户；新写入立即受外键检查。进入受控试运行前必须治理历史 actor，并显式执行 `VALIDATE CONSTRAINT`。历史导入批次使用迁移占位 payload hash，不允许被当成可确认的新批次。

## 3. Unit of Work 与锁策略

`PostgresQuoteUnitOfWork` 使用单个 `engine.begin()` 包住业务写入、状态变化和 S2 审计事件。异常自动回滚；Repository 只使用该事务连接，不在内部另行提交。

锁顺序固定为：采购事项 → 导入批次 → 审批记录。当前动作按需要执行 `SELECT ... FOR UPDATE`：

| 动作 | 事务与锁 | 冲突行为 |
|---|---|---|
| 导入确认 | 锁事项及批次；重读状态、payload hash、完整性、币种和可编辑状态；批量写入后确认批次并写审计 | 同批次并发只允许一个成功；其余 409；任何阻断错误零行业务数据落库 |
| 人工推荐 | 锁事项；从数据库重读 3×5 明细并重新计算排名；不接受客户端排名/最低价 | 非最低价缺理由或审批意见时整体回滚；推荐与审计同事务 |
| 提交审批 | 锁事项；重算完整性、币种、推荐规则；创建 pending 审批、更新事项和审计 | 部分唯一索引保证每事项最多一个 pending；冲突返回 409 |
| 批准/退回 | 先读取当前审批 ID，再在决定事务锁事项与审批记录；校验 pending 和提交人 | 两审批人并发只有一个更新成功；另一个 409；提交人自审 403；决定、事项状态和审计同事务 |
| 版本预留/完成 | 预留事务锁事项并执行计数器 `UPDATE ... RETURNING`；完成事务锁事项并将版本从 reserved 改为 ready | 不使用 `MAX(version)+1` 或内存长度；版本号不会重复或回收 |

SQL 唯一/外键冲突统一转换为 409 `concurrent_write_conflict`；已确认批次、已关闭审批、非 reserved 版本使用明确的 409 业务错误。

## 4. 导入、计算与人工边界

1. 预检把 Items、Suppliers、Offers 解析为规范化 JSON，并对规范化 payload 计算 SHA-256。
2. 确认时不信任预检时的进程内对象，重新从数据库读取批次并重算 hash。
3. 导入有缺报价、币种不一致、数量不正、金额为负、重复供应商+品项等阻断错误时不得确认写入。
4. 行总价继续由 `数量 × 单价` 计算；供应商级运费和税费分别只加一次；数据库读取后实时聚合 subtotal/total，重启后无需依赖进程缓存。
5. 系统只输出最低价排序；`recommended_supplier_id` 默认空白，必须人工选择。非最低价推荐必须同时填写理由和审批意见。
6. 不自动选供应商、审批、下单、发信；AI/OCR、DeepSeek、供应商搜索均未接入。

## 5. Excel 追加式版本流程

正式导出严格拆分为：

1. **版本预留**：数据库事务原子递增 `next_export_version`，插入 `reserved` 版本及 `quote.export_reserved` 审计。
2. **临时生成**：使用 `quote_id + version + UUID` 的临时文件路径生成 Excel。
3. **文件校验**：以 openpyxl 重新打开，计算 SHA-256 和字节大小。
4. **不可覆盖发布**：在同一受控文件系统内使用 `os.link` 实现 put-if-absent 发布；目标存在即失败，绝不覆盖历史文件。
5. **完成登记**：发布成功后，在新数据库事务把版本从 `reserved` 更新为 `ready`，持久化 hash/size/published_at，更新事项状态并写 `quote.export` 审计。

失败规则：

- 生成、校验或发布失败：版本标记 `failed`，保存失败原因并写失败审计；版本号不回收，下一次使用新版本号。
- 文件已发布但 ready 登记失败：数据库保留 `reserved`；受控管理员可运行 `scripts/reconcile_s2_exports.py` 检查现有文件。文件可读则补记 ready 和对账审计；缺失/损坏则记 failed 和失败审计。
- 对账不删除、不覆盖文件；下载只允许 `ready` 元数据，且 storage key 必须解析在配置根目录内。
