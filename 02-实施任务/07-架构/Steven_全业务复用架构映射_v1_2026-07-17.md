# Steven 全业务复用架构映射 v1

- 日期：2026-07-17
- 目的：说明 D0 底座如何持续承载 Steven 的 S1、S2、S3，而非形成一次性 Demo 旁路
- 当前真实实现：仅 S2 报价扫描导入切片
- 当前未实现：S1 文书业务、S3 库存业务及其 OCR/AI 页面、Schema、确认器和正式表写入

## 1. 共用底座

S1/S2/S3 必须复用同一套：

- 认证、Session、CSRF 与 permission-based RBAC。
- PostgreSQL Repository 与 Unit of Work。
- `files` 文件元数据和受控文件存储接口。
- `ocr_jobs` 与 `ai_jobs`。
- `review_candidates` 候选、人工修订与状态机。
- evidence 页码、原文、bbox 与置信度。
- `request_id`、平台审计与业务审计关联。
- approvals 审批底座。
- 版本化导出、追加式历史文件与可恢复一致性 Saga。

后续模块只能追加模块表、Schema、业务确认器和规则，不得另造第二套文件、任务、候选、审计、审批或版本基础设施。

## 2. 通用处理链

```text
受控文件上传
  → files
  → ocr_jobs
  → ai_jobs
  → review_candidates(needs_review)
  → 人工修订/拒绝/确认
  → 模块确认器 + Unit of Work
  → 模块正式表 + 业务审计
  → 人工审批
  → 版本化导出
```

强制规则：

- OCR/AI 永远只生成候选。
- 人工确认前不得写正式业务表。
- 确认时必须重新运行模块业务规则，不能信任客户端或 AI 的金额、排名、状态和决策。
- AI 永远不得自动推荐供应商、审批、下单、发信或确认库存。

## 4. S1 后续业务差异

S1 需要按阶段新增：

- 模板与变量定义。
- 事项、预算、日期、地点、供应商、特殊条款。
- 截止日期不得早于生成日期加 3 天。
- 预算非负且下限不高于上限。
- 供应商名称不得重复。
- 未替换 `{{...}}` 时标记 `draft_error`。
- `draft_error` 或未审批不得正式导出。
- Word 草稿、人工复核、审批、正式导出版本。

AI 可参与：

- 从脱敏来源文件提取待复核字段。
- 生成待复核条款草稿。

AI 不可参与：

- 决定模板是否启用。
- 决定供应商、预算、审批结果。
- 跳过未替换变量和人工复核。

## 5. S2 当前实现与后续差异

当前 D0 已实现：

- 报价文件上传。
- OCR/AI Adapter 合同。
- 严格 `steven.s2.quotation` Schema。
- evidence 与 `needs_review` 候选。
- 人工修订、确认、拒绝。
- 事务确认后写入正式报价。
- 3 家供应商 × 5 品项的 HKD 脱敏基线。
- 既有规则比价、人工推荐/审批和版本化 Excel 能力复用。

后续仍需：

- 在获批 PostgreSQL 环境执行真实 migration、重启恢复与并发验证。
- 在获批网络/密钥环境执行 Azure/DeepSeek 联通和失败演练。
- 若启用 `quote_exception_explanation`，只输出解释候选，不改变规则结果。

## 6. S3 后续业务差异

S3 需要按阶段新增：

- SKU、品项名称、库位、账面量、盘点量、安全库存、目标库存。
- 盘点量、安全库存、目标库存为非负整数。
- 负数库存和重复 SKU 阻断保存。
- `建议订货量 = max(0, 目标库存 - 本次盘点量)`。
- 盘点量低于安全库存时标红。
- 实物盘点与订货结果必须人工确认。

AI/OCR 可参与：

- 从盘点 Excel/PDF/图片提取待复核候选。
- 解释低库存、差异或字段异常。

AI 不可参与：

- 确认实物库存。
- 自动确认补货。
- 自动下单。

## 7. PostgreSQL、审计与导出复用

### PostgreSQL

- 使用既有环境隔离、migration 链、Repository/UoW 模式。
- D0 通用表不复制到各模块。
- 模块正式写入由独立 confirmer 在事务内完成。
- S1/S3 只追加 revision，不回写 `0001`–`0007`。

### 审计

- 平台事件继续写平台审计仓。
- 各模块业务写入与业务审计保持同一事务。
- 文件、job、candidate、审批、导出均以 `request_id` 与 object ID 关联。
- 普通 Steven 用户不能删除审计或读取全量平台审计。

### 导出

- S1 Word、S2 Excel、S3 Excel 共用版本预留、临时生成、校验、原子发布、ready/failed 登记与恢复巡检原则。
- 版本号不可回收或复用。
- 历史文件不可覆盖或删除。
- 文件系统与数据库之间采用可恢复一致性 Saga，不宣称跨资源原子事务。

## 8. 架构防旁路约束

后续实现不得：

- 在 S1/S3 内直接调用 Azure/DeepSeek 并写正式表。
- 为每个模块另建 OCR 状态机。
- 将 candidate JSON 当作已核验业务数据。
- 在前端计算并提交可信金额、排序、权限或审批状态。
- 绕过现有认证、CSRF、RBAC、审计和 Unit of Work。
- 将 Mock、缓存或人工 ground truth 标记为实时线上结果。

## 9. 当前准确结论

- 通用文件、OCR、AI、候选、证据、人工确认、审计关联和 PostgreSQL migration 合同已经建立。
- S2 扫描导入切片已完成代码与离线验证。
- S1/S3 尚未实现，只能在后续阶段按本映射扩展。
- 真实 PostgreSQL、Azure、DeepSeek 均未连接或验证。
